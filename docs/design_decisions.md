# 기술적 설계 결정 (Architecture & Design Decisions)

본 문서는 L2C 프로젝트를 진행하며 논의되고 적용된 핵심 설계 의사결정(Design Decisions)과 그에 따른 트레이드오프(Trade-offs)를 기록한 문서입니다.

---

## 1. 하이브리드 수집 구조의 채택: Classic DOM 파싱 vs SoM 비전 에이전트

### [결정] 두 방식의 엔진을 모두 구현하고 상호 비교 벤치마크 수행
프로젝트 초기에는 한 가지 방식에만 집중하려 했으나, 실무 관점에서의 정량적 정밀도와 비용 효율성을 비교 검증하기 위해 두 트랙을 모두 병렬 구현했습니다.

*   **Classic DOM 스크래퍼 (Playwright 기반)**:
    *   **장점**: 매우 빠르고(건당 수 초 내외), API 비용이 들지 않으며, 구조화된 데이터를 100% 정밀도로 파싱할 수 있음.
    *   **단점**: 프론트엔드 마크업 구조가 조금만 바뀌어도 수집기가 붕괴(Fragile)되며, 사이트별로 전용 어댑터를 계속 작성해야 함.
*   **SoM 비전 에이전트 (LangGraph + YOLOv8 + PaddleOCR + VLM)**:
    *   **장점**: DOM 트리에 전혀 의존하지 않으며, 사람이 눈으로 브라우저를 조작하는 것과 같은 방식으로 제로샷 일반화(Zero-shot Generalization)가 가능함. 사이트 개편에 절대적으로 강인함.
    *   **단점**: VLM 호출 토큰 비용 및 레이턴시가 발생함.
*   **결과**: 본 프로젝트는 두 시스템을 모두 구현하여 Jaccard Similarity 기준 정합성을 비교 벤치마킹하도록 설계했습니다. 이는 Phase 7의 **하이브리드 RAG 에이전트**를 위한 초석이 되며, 비전 에이전트가 수집한 미가공(Raw) 데이터와 구조화된 데이터의 크로스 검증을 가능하게 합니다.

## 2. Set-of-Marks (SoM) 메커니즘의 선택 및 좌표 직접 예측 배제

### [결정] VLM에게 화면의 절대 좌표(예: `{"point": [350, 420]}`)를 출력하도록 요구하는 대신, 화면에 숫자 마커(Mark)를 합성하여 마커 ID를 선택하도록 유도

*   **이유 (거부한 대안 - 좌표 직접 예측)**:
    - VLM(Gemini, GPT-4o 등)은 이미지 내 객체의 픽셀 좌표를 정확히 예측하는 능력이 저하됩니다. 화면 해상도나 DPI 배율에 따라 오차가 커져 엉뚱한 곳을 클릭하는 클릭 미스(Click Miss)가 빈번했습니다.
    - 또한 좌표를 문자열로 디코딩하고 다시 물리 마우스 이벤트로 변환하는 프롬프팅 과정에서 토큰 소모와 오동작이 잦았습니다.
*   **해결책 (SoM 채택)**:
    - 로컬 디텍터(YOLOv8로 아이콘 검출 + PaddleOCR로 텍스트 영역 검출)를 통해 클릭 가능한 후보군 바운딩 박스를 물리적으로 100% 정확하게 따냅니다.
    - 각 영역에 고유 숫자 태그(`[0]`, `[1]`, `[2]` 등)를 합성한 이미지를 VLM에 전달합니다.
    - VLM은 단순하게 `"마커 3번을 클릭하라"`는 이산적(Discrete) 의사결정만 내리며, 에이전트 드라이버가 3번에 매핑된 정확한 중심점 물리 좌표로 `pyautogui` 클릭을 실행합니다. 이로써 **VLM의 공간 지각 한계를 로컬 CV 엔진의 정확성으로 완벽히 보완**했습니다.

## 3. 거부된 대안 기술들 (Rejected Alternatives)

### ① Anthropic Computer Use API
*   **거부 이유**: 
    - Anthropic의 OS 제어 API는 화면 전체 스크린샷을 전송하고 OS 마우스/키보드를 직접 다루기 때문에 지연 시간(Latency)이 매우 길며 API 단가가 극도로 높습니다.
    - 또한 국내 채용 사이트(원티드, 리멤버 등)의 한국어 텍스트 레이아웃 및 팝업 대처율이 상대적으로 떨어졌습니다.
    - 비용 통제와 로컬 성능 최적화를 위해, 브라우저 스크린샷 캡처 영역을 제어하고 오픈소스 YOLOv8/PaddleOCR 레이어를 직접 호스팅하는 로컬 SoM 파이프라인을 구축했습니다.

### ② 풀페이지 VLM 직접 입력 및 파싱 (Pure Screenshot-to-JSON)
*   **거부 이유**:
    - 브라우저 화면 전체 스크린샷 하나만 VLM에 던져주고 바로 JD 데이터 JSON을 추출하도록 하는 방식입니다.
    - 언뜻 간단해 보이지만, Wanted 등의 채용 공고는 본문이 길어서 스크롤을 내려야 전체 텍스트가 노출됩니다. 한 번의 캡처로는 전체 텍스트를 유실 없이 수집할 수 없습니다.
    - 결과적으로 '스크롤을 내리며 텍스트를 점진적으로 누적 수집하는 루프' 제어권이 포함된 에이전트 구조가 필수적이었습니다.

## 4. CUDA DLL 충돌 우회를 위한 프로세스 격리 및 최적화

### [결정] YOLOv8(PyTorch)과 PaddleOCR(PaddlePaddle)의 단일 런타임 충돌을 회피하기 위해 CPU/GPU 메모리 상에서 독립 구동하고, OCR을 서브프로세스로 분리 실행

*   **이유**: Windows 환경에서 두 딥러닝 프레임워크가 동일 프로세스 안에서 CUDA 컨텍스트와 DLL 기호를 함께 로드할 경우 기호 탐색 오류(`WinError 127`)가 나며 프로세스가 크래시됩니다.
*   **최적화 기법**:
    - `paddle_ocr_runner.py`를 독립 서브프로세스로 분리하여 `subprocess.run`으로 실행합니다.
    - 서브프로세스 진입 시 `sys.modules['torch'] = MagicMock()` 모킹 패치를 가하여 PyTorch가 임포트되어 충돌하는 현상을 원천 방지합니다.
    - PaddleOCR 초기 기동 시 7초 이상 소요되던 분석 지연을 막기 위해 `paddle.inference.Config.switch_ir_optim = False` 패치를 적용, 내부 연산 최적화 단계를 스킵함으로써 기동 속도를 **0.35초**로 단축했습니다.
