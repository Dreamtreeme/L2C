# L2C 에이전트 트러블슈팅 및 연대기 (Chronological Debugging Log)

본 문서는 L2C 비전 에이전트를 개발하는 과정에서 겪은 수많은 시행착오, 사용자와의 실시간 피드백 루프, 로그 분석, 가설 수립 및 검증 과정을 **시간 순서(Chronological Storytelling)**로 기록한 개발 비사입니다.

---

## 🎬 1막: 첫 E2E 테스트와 로그인 굴레 (Login Loop)

### [문제 발견] 바탕화면에서 시작한 에이전트의 정체
* **목표**: "현재 바탕화면입니다. 원티드 사이트에 접속하여 로그인한 뒤, '데이터 분석가' 채용 공고를 검색해 주세요."
* **초기 상태**: LangGraph 워크플로우에 `Perception Node` (로컬 Qwen2.5-VL 3B via Ollama)와 `Reasoning Node` (Gemini 3.5 Flash)를 연결하여 실행했습니다. 에이전트는 기특하게 브라우저를 켜고 원티드 메인 페이지까지 진입했으나, '회원가입/로그인' 버튼을 클릭한 뒤 일반 로그인 아이디/비밀번호 폼을 채우려는 시도를 반복하며 로그인 모달창 주위를 겉돌았습니다.

### [사용자 질문 및 피드백]
> *"보니깐 무조건 로그인 여부를 확인하려 하는데 자동로그인 되어있을 수 도 있으니 해보고 로그인이 필요하면 로그인 하는걸로.*
> *그리고 로그인 일단 구글 로그인 클릭해서 현재 등록되어있는 구글계정으로 클릭해서 로그인하게"*

### [로그 분석 후 가설 및 조치]
* **상태 분석**: 지휘자(Gemini) 프롬프트에 로그인 상태에 대한 사전 판단 지침이 없었으며, 번거로운 일반 로그인 폼 입력 대신 브라우저 세션에 남아있는 구글 간편 로그인(Google Sign-In) 연동 로직을 활용할 설계가 부재했습니다.
* **해결법 시도**: 
  - [commander.py](file:///c:/Users/psg/Desktop/L2C/agent/prompts/commander.py) 프롬프트에 **"로그인 여부 판단 및 간편 로그인 가이드라인"**을 추가했습니다.
  - 자동 로그인 여부를 먼저 확인하여 건너뛰도록 지시하고, 로그인이 필요하면 반드시 **"Google로 계속하기"**를 먼저 누르도록 고정했습니다.
  - 구글 계정 선택 창이 뜨면 등록된 구글 이메일 요소를 정확히 클릭하도록 구체화했습니다.

## ⚡ 2막: JSON 포맷의 대붕괴와 무한 루프

### [다른 문제 발견] 터미널에 쏟아지는 개행 문자의 파도
* 구글 로그인 버튼을 클릭하기 위해 로컬 VLM을 호출하는 순간, Ollama 서버가 굳어버리거나 Python JSON 파서가 크래시를 일으켰습니다. 터미널에는 `\n\n\n\n...` 또는 `0,0,0...` 처럼 의미 없는 빈 값만 무한히 응답되는 현상이 잡혔습니다.

### [사용자 가설 및 지시]
> *"Ollama 호출 시 format="json" 옵션을 강제하고, 프롬프트에 "반드시 단일 JSON 배열로만 응답하라"고 명시.*
> *그다음, 파서를 정규식 매칭이 아니라 응답에서 모든 {...} 객체를 수집해 배열로 재조립하는 방식으로 보강.*
> *Reasoning 노드에 루프 감지 로직을 추가해줘."*

### [로그 분석 후 가설]
* **분석**: Ollama의 `format="json"` 제약은 추론 엔진의 샘플링 확률을 강제로 제어합니다. 경량 VLM(3B)의 좁은 Attention 연산량 한계 속에서 "시각 좌표 계산"과 "문법 준수"라는 두 무거운 작업을 병렬로 처리하려다 보니 내부 연산 상태가 완전히 붕괴되어 무한 공백 출력 루프(Degeneration)에 빠진 것입니다.

### [해결법 시도]
* 문법 에러가 나더라도 구조를 구제할 수 있도록, 텍스트에서 괄호 `{`와 `}`의 대칭을 스택(Stack)으로 추적하여 유효한 JSON을 조립하는 **Stack-based Parser**를 `perception.py`에 이식했습니다.
* 3회 연속으로 동일한 버튼을 클릭하는 루프가 감지되면 LLM에게 경고를 보내고 작동을 중단하는 **루프 감지 차단기**를 `nodes.py`에 긴급 장착했습니다.

## 🎯 3막: 마우스 포인터의 방황과 고정밀 Perception 전환

### [다른 문제 발견] 돋보기를 누르랬더니 하늘을 찌르는 마우스
* 포맷 에러는 잡았으나, 이번에는 마우스 포인터가 GNB 바의 '돋보기(검색)' 아이콘을 누르지 못하고 화면 엉뚱한 한가운데를 클릭하는 현상이 발견되었습니다. 에이전트는 계속 헛발질 클릭을 하며 무한 루프 경고에 걸렸습니다.

### [사용자 질문 및 검증 요청]
> *"좌표를 잘 클릭 못하는데 어떤 해결책들이 있는지 제안"*
> *"가설이 맞는지 좌표 로그 검증해줘"*
> *"계속 이상한데 클릭하는데 로그 확인해봐"*

### [로그 분석 후 가설 및 실험]
* **로그 역추적**: 마커 좌표 로그를 검증해 보니, 원래는 약 `(1440, 227)` 부근에 위치해야 할 돋보기 아이콘의 좌표를 로컬 VLM이 `(1050, 500)` 같은 엉뚱한 중앙 좌표로 매핑하고 있었습니다.
* **가설 1**: VRAM 부족으로 이미지 해상도를 `512px`로 압축 구동했는데, 이 과정에서 돋보기 아이콘이 고작 2~3픽셀 크기로 뭉개져 VLM의 공간 정합성(Grounding) 능력이 무력화되었습니다.
* **가설 2**: 3B 파라미터 경량 모델 자체의 시각적 공간 인지 한계입니다.
* **실증 실험**: Ollama(Qwen-VL 3B)의 `512px` 캡처 데이터 좌표 매핑 오차율이 60%를 상회함을 로그 분석으로 입증했습니다.

### [해결법 시도]
* 로컬 3B VLM의 공간 인지 한계를 극복하기 위해, **Perception Node의 시각 분석기에 'Gemini 3.5 Flash API'를 메인으로 투입**하는 대전환을 감행했습니다.
* Gemini의 정밀한 시각 해석 능력 덕분에 돋보기 아이콘 좌표가 `(1444, 227)`로 완벽히 복구되어, 마우스 포인터가 돋보기 아이콘 정중앙을 정확하게 타격하기 시작했습니다.

## 🔍 4막: 8개 탐지 한계에 가려진 검색 결과와 무한 루프

### [다른 문제 발견] 검색어 입력은 성공했는데 왜 끝내질 못하니
* Gemini Perception 덕분에 구글 계정을 정확히 눌러 로그인을 완료하고, 돋보기를 눌러 검색창에 '데이터 분석가'를 clipboard 복사-붙여넣기 방식으로 완벽히 입력한 뒤 추천 태그 클릭까지 성공했습니다.
* 브라우저 화면에는 '데이터 분석가 채용 공고' 리스트가 가득 떴는데, 에이전트는 다음 동작을 결정하지 못하고 머뭇거리다 `recursion_limit`에 걸려 강제 종료되었습니다.

### [로그 분석 후 가설]
* **최종 UI 컨텍스트 확인**: 에이전트가 마지막으로 전달받은 UI 컨텍스트 로그를 뜯어보았습니다.
  ```
  [id: 0] 데이터 분석가 검색창
  [id: 1] 채용, [id: 2] 이력서, [id: 3] 이벤트, [id: 4] 직군...
  ```
* **원인 발견**: Gemini Prompt에 `"Extract at most 8 elements(최대 8개 요소를 추출하라)"`는 제약 조건이 걸려 있었습니다. 이 때문에 화면 상단의 글로벌 내비게이션 바(GNB) 메뉴가 8개의 슬롯을 전부 독차지했고, 본문 아래쪽에 성공적으로 출력된 **채용 공고 카드들이 인지 리스트에서 아예 배제**된 것이었습니다. 지휘자는 채용 목록을 눈앞에 두고도 보지 못해 태스크 완료 선언(`finish_task`)을 하지 못했습니다.

### [해결법 시도]
* [perception.py](file:///c:/Users/psg/Desktop/L2C/agent/tools/perception.py)의 Gemini UI 요소 검출 제한을 **`25개`**로 크게 확장했습니다.
* 다시 시뮬레이션을 돌리자, 드디어 비주얼 컨텍스트 하단에 `[id: 9] [신입] 데이터 분석가 채용` 등의 본문 카드가 정상 검출되었습니다. 지휘자는 이를 인지하고 즉시 `finish_task`를 선언하며 **E2E 시나리오를 완벽하게 정식 성공**시켰습니다!

## 🔋 5막: 0원 로컬 구동(7B)의 한계 성능 테스트

### [사용자 질문]
> *"비용 문제를 로컬에서 돌리면 0원이라 가정하고 외부 모델 파라미터 성능 감안해서 얼마나 차이있었는지 계산"*
> *"모델을 더 큰걸 써볼수없나? 양자화 기능까지 쓰면 더 큰거 쓸수있을거같아"*
> *"qwen2.5vl:7b 있으니 바꿔서 해줘"*

### [로그 분석 및 가설 검증]
* 사용자님의 PC에 탑재된 **RTX 3080 (10GB)** 그래픽 카드를 인식하고 가용 VRAM 용량을 계산했습니다.
* Q4 양자화된 `qwen2.5vl:7b` 모델(4.7GB)과 1024px 이미지 KV 캐시(2GB)는 백그라운드 사용량을 감안하면 아슬아슬하게 GPU 내에 안착(총 8~9GB 점유)할 수 있음을 확인했습니다.
* 로컬 모델 세팅을 7B로 올리고, 해상도를 `1024px` 및 `768px`로 다르게 조정한 후 **로컬 한계 성능 시뮬레이션**을 실시했습니다.

#### 🧪 실험 결과 및 지표 대조
1. **7B (1024px, num_ctx: 4096)**: **43.18초** 소요. 모든 요소 완벽 검출 및 클릭 좌표 오차 0px에 수렴 (대성공).
2. **7B (768px - 가설 1)**: **46.64초** 소요. 해상도 하향으로 인해 좌표가 왼쪽으로 356px 빗나가며 클릭에 실패하는 정합성 붕괴 현상 재발 (기각).
3. **7B (1024px, num_ctx: 2048 - 가설 3)**: **54.81초** 소요. VRAM을 1GB 아꼈으나 속도는 단축되지 않음.
4. **대조군 3B (768px)**: **7.90초** 소요. 속도는 매우 빠르나 인지 과부하로 요소 1개만 검출.

* **최종 병목 확인**: 7B 모델 구동 시 50초 가까이 걸리는 진짜 범인은 윈도우 환경 Ollama 백엔드가 비주얼 어텐션 연산 시 **Flash Attention 2 가속을 컴파일하지 못해** 발생하는 비주얼 프리필(Prefill) 병목이었습니다.

## 6막: 또 다른 도전 - Hugging Face 직접 구동으로의 우회

### [사용자 질문]
> *"Ollama를 안쓰고 허깅페이스에서 직접 모델을 다운받아서 써도 똑같을까?"*

### [결론 및 차기 마일스톤 (Phase 3.5 신설)]
* **똑같지 않습니다.** Ollama라는 포장지를 벗겨내고 허깅페이스에서 모델을 직접 내려받아 Python PyTorch 환경에서 구동하면:
  1. 윈도우 환경에 `flash-attn` 라이브러리를 직접 컴파일 연동하여 **비주얼 어텐션 연산을 3~4배 가속**시킬 수 있습니다.
  2. `max_pixels` 파라미터를 통해 해상도 손실 없이 영리하게 이미지 토큰만 압축할 수 있습니다.
  3. 로컬 7B 추론 속도를 **10초 이내**로 정렬하는 쾌적한 "0원 에이전트" 구현이 가능해집니다.
* 이 도전 과제를 L2C 프로젝트의 차기 중간 계획인 **`Phase 3.5`**로 정식 신설하고 리드미에 등록하며 오늘의 긴 여정을 갈무리했습니다.
